# azf-project-note-binding

这个 skill 用来把代码项目和 Obsidian 项目笔记绑定起来，并维护 `4_项目理解` 这一层：代码理解索引、源码拆解、流程理解、代码变更同步记录和 Excalidraw 结构图。

## 什么时候触发

- 用户要求“项目和笔记绑定”“代码改了笔记怎么同步”“同步项目笔记”。
- 新项目需要建立 `.azf/project-notes.yaml`。
- 代码理解、源码拆解、代码结构分析、文件结构图或项目理解总图需要沉淀到 Obsidian。
- 需要为代码/项目生成 Excalidraw 可视化结构图。

## 当前关键规则

- 绑定文件 `.azf/project-notes.yaml` 是项目笔记位置的事实来源。
- 本 skill 只负责 `4_项目理解`；`0_总览`、`1_主线笔记`、`2_调试记录` 等整体工作记录结构由 `azf-obsidian-work-record` 负责。
- 新项目工作记录初始化时，只有用户确认需要代码拆解/项目理解，才启动本 skill；如果不需要，`4_项目理解/` 先空着。
- 新项目默认把代码理解放入 `4_项目理解/`，根入口使用 `0_项目理解总图.excalidraw.md`、`1_项目理解索引.md`、`2_最低限度源码拆解.md`。
- 实验记录不再作为项目根目录结构，默认放在 `2_调试记录/实验记录/`；代码同步记录放在 `4_项目理解/代码变更同步记录/`。
- 代码变化后同步笔记前，除非用户已经明确授权，否则先问是否同步。
- Excalidraw 图要用无限画布思维，卡片尺寸根据内容估算，保留空白，不要为了单屏截图强行压缩。
- 代码结构图的真实处理卡片要在代码框右下角保留淡色小字的数据痕迹，例如 `cfg + .mat -> MatDatasetArrays`；空间不够时扩大代码框和卡片，不要省略或挤压。
- 视觉图生成后要截图或渲染预览，检查文字裁切、重叠、箭头穿过内容、连线交叉、虚线箭头贴边/重叠、箭头压卡片上沿和卡片拥挤问题。
- 涉及深度学习实验记录时，配合 `azf-deep-learning-experiment-record`。

## Excalidraw 画图避坑

- 不要把无限画布压成单屏 PPT；优先保证局部清楚、节点之间有留白。
- 不要用巨大的虚线阶段框硬套结构；容易错位、显乱。阶段可以用小标签或有意义的虚线卡片表达。
- 卡片内容只写代码理解：中文角色名、脚本/函数名、简短职责。不要写“子图已连接”“已同步”这类维护状态。
- 每张卡片不要重复塞多个链接；通常一个底部跳转入口就够。
- 箭头必须绑定节点，路线要手绘、清爽、避开所有文字和代码框；虚线箭头也按完整箭头验收，不允许贴着或重叠卡片边框、填充纹理和其他连线；不要为了最短路径穿过内容，也不要让连线互相交叉或直接贴着/跨过卡片上沿。
- 卡片必须按标题、代码、说明和右下角淡色数据痕迹自适应尺寸；空间不够就加大，不要挤压。
- 文件级代码学习图不能省略大量源码；太长就拆成多张图或主干加分支。
- 完成前必须截图/渲染检查，不只看 JSON：重点看重叠、裁切、箭头、卡片平衡和整体是否仍有清新手绘感。
