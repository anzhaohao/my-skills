# Paper Obsidian Review

## 目标

把论文、代码仓库、本地 PDF 和在线资料整理成可直接放入 Obsidian 的中文论文调研笔记。输出应适合后续写项目依据、文献综述、复现计划和指标支撑材料。

核心原则：不要写泛泛的“这篇工作干了什么”。必须明确“为什么做、解决什么问题、用什么方法解决、创新在哪里、如何复现、与用户指标如何关联”。

## 触发场景

使用本 skill 当用户要求：

- 总结论文、代码、相关工作、论文调研、文献综述。
- 把论文内容放进 Obsidian、vault、Web Clipper、Markdown 笔记。
- 需要整理本地 PDF、本地代码、在线论文、在线代码链接。
- 需要提取论文方法图、框架图、系统图。
- 需要判断代码复现可行性。
- 需要把多篇论文按统一结构对比。

## 输入信息收集

优先向用户确认或从本地发现以下信息：

- Obsidian vault 路径，例如 `E:\obsidian\human-robot`。
- 本地代码根目录，例如 `C:\code\human-robot`。
- 本地 PDF 目录，例如 `D:\论文\人机交互`。
- 每篇论文的在线论文链接、在线代码链接、项目主页。
- 用户关心的指标或主题，例如“大模型驱动的人机混合增强智能”“多模态人机交互”。

如果用户未给 vault 路径，先寻找 `.obsidian` 文件夹；如果找不到，指导用户在 Obsidian 主程序中创建/打开 vault。不要把浏览器插件 Obsidian Web Clipper 当作 Obsidian 主程序。

## 推荐工作流

### 1. 盘点材料

- 列出代码目录中的项目名、README、依赖文件、入口脚本、图片文件。
- 列出 PDF 目录中的 PDF 文件名。
- 对每篇论文建立映射：论文名 -> 本地 PDF -> 本地代码 -> 在线论文 -> 在线代码。
- 对不确定映射明确标注“待确认”，不要猜错链接。

### 2. 建立 Obsidian 结构

默认把单篇笔记放在 vault 根目录，除非用户要求子目录。

推荐目录：

```text
<vault>/
  CoCoDial.md
  FAM-HRI.md
  TCC-IRoNL.md
  三篇人机机器人交互论文对比.md
  attachments/
  pdfs/
```

如果用户明确“不需要多层目录”，不要创建 `Papers/Human-Robot` 这类嵌套目录。

### 3. 处理 PDF 链接

优先把 PDF 复制到 vault 内部 `pdfs/` 文件夹，然后使用 Obsidian 内部链接：

```markdown
- 本地 PDF：[[pdfs/文件名.pdf]]
```

不要优先使用绝对路径 Markdown 链接，因为 `D:/中文路径/带空格.pdf` 在 Obsidian 中可能无法直接打开。

### 4. 处理论文方法/框架图

只保留“论文方法图、框架图、系统图、pipeline 图”。不要把统计图、结果图、环境演示图、导航示例图当作方法图，除非用户明确要求。

推荐放入：

```text
<vault>/attachments/<paper>-method.png
```

并在笔记中引用：

```markdown
## 论文方法/框架图

![[attachments/<paper>-method.png]]
```

若本地代码没有图片，可从 README、项目主页或论文 PDF 中提取。若无法确认是哪张方法图，先说明候选图并让用户确认。

### 5. 单篇笔记结构

每篇论文使用以下结构。标题必须学术化、清晰，不要口语化。

```markdown
---
title: "..."
source: "..."
code: "..."
tags:
  - paper
  - ...
created: YYYY-MM-DD
---

# 论文简称 / 系统名

## 基本信息

论文题目：...

## 链接

- 本地笔记：`...`
- 本地 PDF：[[pdfs/xxx.pdf]]
- 本地代码：`...`
- 在线论文：...
- 在线代码：...
- 项目主页：...

## 论文方法/框架图

![[attachments/xxx-method.png]]

## 研究背景与问题动因

说明该方向为什么需要这篇工作。写研究动因，不写泛泛宣传。

## 核心科学问题

用“核心问题 + 子问题”写清楚论文要解决什么。

## 问题求解路径

逐条写明：论文使用什么方法，解决什么问题。
格式推荐：

- 使用 X，解决 Y 问题。
- 使用 A+B 融合，解决 C 的不确定性问题。
- 构建 D 数据集/系统，解决 E 缺少基准或落地链路的问题。

## 方法定位

说明该方法属于什么范式、系统、任务或模型类型，在相关研究链条中的位置是什么。

## 输入输出

输入：...

输出：...

## 方法框架

按模块写清楚系统组成和数据流。

## 关键机制

提炼论文真正起作用的机制，例如轨迹规划、凝视融合、LLM 指令解析、ROS 执行映射等。

## 主要创新贡献

列出 3-6 条，强调相对已有工作的新增点。

## 数据集设计 / 代码结构 / 配置依赖

按论文类型选择一个标题：

- 数据集论文或 benchmark：用 `## 数据集设计`。
- 系统/机器人代码：用 `## 代码结构` 或 `## 配置依赖`。
- 训练算法论文：可用 `## 实验设置与训练配置`。

## 复现要点

说明入口脚本、运行方式、环境依赖、模型权重、API key、硬件要求和主要卡点。

## 与指标关联

把论文明确对应到用户的项目指标、课题指标或研究主线。
```

### 6. 多篇对比笔记结构

创建一篇总览笔记，用于横向比较：

```markdown
# 多篇论文对比

## 对比表

| 工作 | 主要对象 | 输入模态 | 输出/目标 | 适合支撑的研究点 | 复现难度 |
|---|---|---|---|---|---|

## 链接汇总

| 工作 | 本地笔记 | 本地 PDF | 本地代码 | 在线论文 | 在线代码 |
|---|---|---|---|---|---|

## 总体理解

用一段话串起多篇工作的研究链条。

## 与项目指标的对应关系

分指标说明哪篇最直接、哪篇间接支撑。

## 复现优先级建议

给出先复现哪篇、为什么、卡点是什么。
```

## 复现可行性判断

评估复现时至少检查：

- README 是否完整。
- 是否有 requirements、environment、Dockerfile、docker-compose。
- 是否需要 ROS、Gazebo、CUDA、特定设备、模型权重、API key。
- 是否有数据集、预训练权重、运行脚本。
- 当前机器/服务器是否具备硬件和系统环境。

机器人/多模态项目常见判断：

- 仅有 Python 脚本和数据：较容易。
- 需要外部大模型 API：中等，取决于 key 和 base_url。
- 需要 ROS/Gazebo/Docker/GPU：中高。
- 需要专用设备，例如 Project Aria：完整复现较难，可做模块级复现。

## 标题规范

优先使用这些学术化标题：

- `研究背景与问题动因`
- `核心科学问题`
- `问题求解路径`
- `方法定位`
- `输入输出`
- `方法框架`
- `关键机制`
- `主要创新贡献`
- `数据集设计`
- `代码结构`
- `配置依赖`
- `复现要点`
- `与指标关联`

避免：

- `这篇工作干了什么`
- `简单介绍`
- `大概内容`
- `我的理解`，除非用户明确要个人化笔记。

## 常见坑

- 不要把 Web Clipper 插件当成 Obsidian 主程序；插件不能单独作为 vault。
- 不要用绝对本地 PDF 链接作为首选；复制进 vault 后用内部链接。
- 不要擅自创建多层目录；先尊重用户指定的 vault 结构。
- 不要把结果图、统计图当方法框架图。
- 不要猜测 PDF 与论文的对应关系；不确定就标待确认。
- 不要只写动机或摘要；必须写清“方法解决问题”的映射。
- Windows 路径中有中文或空格时，Obsidian 内部链接比 Markdown 绝对链接更稳。

## 输出检查

完成前检查：

- 三类链接都能在笔记中看到：本地 PDF、本地代码、在线论文/代码。
- PDF 已在 `pdfs/` 下，图片已在 `attachments/` 下。
- 每篇都有方法/框架图，且不是统计/结果图。
- 每篇都有“研究背景与问题动因、核心科学问题、问题求解路径、主要创新贡献”。
- 每篇保留“方法定位、输入输出、方法框架、关键机制、复现要点、与指标关联”。
- 总览表中的链接和单篇笔记保持一致。
