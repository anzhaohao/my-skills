# azf-obsidian-work-record

这个 skill 用来维护安钊锋的 Obsidian 项目工作记录。它把一个项目整理成“总览、主线、调试记录、硬件信息、项目理解、补充资料”几层，让以后重新接手时能先看懂当前状态，再顺着主线理解每条记录之间的关系。

## 什么时候触发

- 用户要求新开一个项目、建立项目总笔记、整理项目笔记、维护调试记录。
- 项目文件夹在 `01-Project` 下，需要重新拆分、编号、移动或同步链接。
- 用户提供实验截图、GUI 状态图、硬件状态图、报错、结果图，需要写入 Obsidian。
- 用户说“新项目也按这个方式记录”“同步调试记录”“帮我把记录整理清楚”。
- 用户希望新项目从一开始就达到 FrogTrace 这种“总览清楚、主线清楚、调试记录归位、图片不断链”的效果。

## 当前关键规则

- 首选结构：

```text
0_总览/
1_主线笔记/
2_调试记录/
3_硬件信息/
4_项目理解/
5_补充资料/
```

- 总笔记保持轻量，优先作为项目仪表盘；推荐顺序是：当前状态 → 当前主线 / 主线与当前状态 → 目前待解决的问题 / 需求 → 未归类调试记录 / 小 bug 收件箱。
- 新手阅读顺序、常用入口、文件夹说明应迁移到 `0_总览/项目名新手阅读顺序.md` 或补充说明里，不长期堆在总笔记。
- 总笔记正文不重复写文件标题；因为没有正文标题，一级内容标题从 `#` 开始，并加中文序号，例如 `# 一、现在先看这里`、`# 二、当前主线`、`# 三、目前待解决的问题`。
- 总笔记顶部维护 `目前待解决的问题 / 需求` 区域；表格只做轻量索引，不要 `入口` 列，列为：事项、等级、问题名称、状态。
- 事项编号统一用 `Q1`、`Q2`、`Q3`，方便在对话、提交信息和调试记录里直接引用；等级保留醒目的 `🚨 核心`、`⭐ 重要`、`🌱 后续`。
- 问题表格下方固定写一行 `等级可选：🚨 核心 / ⭐ 重要 / 🌱 后续`，方便后续复制调整。
- 问题的描述、当前判断、关联证据和下一步放在表格下面的短卡片里，避免横向表格太挤。
- 总笔记问题表的状态只有两个：`未创建主线`、`已有主线`；新建问题默认 `未创建主线`，已有主线后再改为 `已有主线`。
- 新增问题/需求使用 Templater 命令 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\新增项目问题或需求.md`，只弹出三个输入：问题名称、问题描述、等级选择。总笔记问题表下方放 Buttons 插件按钮，使用 `type cursor template` 调用这个模板，不加 `templater true`。
- 主线笔记用来串联不同工作记录之间的关系，避免单看记录时看不出因果和前后变化。
- 总笔记的主线章节标题下优先放一个轻量 `.base` 索引，只显示三列：创建日期、主线、当前状态，并按创建时间从旧到新排列；base 只做索引，不替代下面手写的主线分组、从属关系和当前判断。主线 Base 模板位于 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\项目主线索引Base模板.base`。
- 总笔记正文模板以 Obsidian Templater 文件为准，当前机器路径是 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\项目总笔记模板.md`。
- 主线笔记正文模板以 Obsidian Templater 文件为准，当前机器路径是 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\项目主线笔记模板-日期推进记录.md`。主线 frontmatter 必须包含 `当前状态: 一句话状态`，供总笔记主线 base 读取；主线名称列由 base 读取 `file.name`，不要另建 `主线` 属性。主线正文使用 `# 一句话说明`、`# 当前状态`、`# 推进记录`、`# 相关入口`。推进记录按日期标题写，不再用 Step 日记；每次推进下面跟随本次方案、调试记录和一句结论。代码状态、gitGraph、命令、日志和证据放进对应调试记录，不写进主线。
- 主线推进记录采用三档分流：一两句话能说清且无证据时写主线并标 `调试记录: 无单独记录`；一段话加说明性截图时用折叠 callout；有证据截图、实机结果、CSV、日志、命令输出、走弯路或未来可能被质疑的结论时必须开调试记录。
- 代码修改、调试、硬件排查、Git 工作和日志阅读期间，Obsidian 笔记库默认只读。只有我明确说“更新笔记、同步笔记、写入笔记、整理笔记”或确认一次笔记同步方案，才允许修改总览、主线、调试记录或硬件卡片。
- 如果代码/调试进展值得记录但我没有要求更新笔记，只在最后说明“笔记未更新，可后续单独同步”，不要顺手改笔记。
- 小 bug 不需要每个都升级成主线，但每个 bug/debug 记录都必须接入入口：先在 `2_调试记录/` 创建记录，并写 `归类状态: 未归类`；总笔记用“未归类调试记录 / 小 bug 收件箱”展示；接入主线后改为 `归类状态: 已归类` 并补 `关联主线`。
- 如果用户明确说“独立记录”“独立调试记录”“单独记录”“不要归类”或类似要求，用户意图优先于自动归类规则：不要因为能找到相关主线就写 `归类状态: 已归类`，默认写 `归类状态: 未归类` 并保持 `关联主线` 为空，除非用户明确要求加引用或接入主线。
- 如果 Obsidian Bases 可用，优先用 `.base` 文件筛选 `项目 == 项目名` 且 `归类状态 == 未归类` 的记录，并嵌入总笔记；如果 Base 格式不确定，先用手动表格。
- 调试记录放实际发生过的过程；目前 FrogTrace 调试记录只使用一种时间线格式：`# 一、调试目标`、`# 二、调试过程`、`# 三、调试结论`，过程里按真实时间顺序写 `Step 1`、`Step 2`、`Step 3`。
- 调试记录正文模板以 Obsidian Templater 文件为准，当前机器路径是 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\项目调试记录模板-Git在Step中.md`。skill 只保留规则和路径，不再内嵌整段模板正文。
- 如果换新电脑、换 vault 位置，或上述模板路径不存在，必须先向我确认当前 Obsidian 笔记库根目录、Templater 模板文件夹和调试记录模板文件名；不要凭猜测在新位置重建模板。
- 不再为调试记录单独区分“实机验证记录”“代码修改记录”“测试表记录”等模板。真实硬件测试、代码修改、复测和日志排查都放进同一种时间线格式里。
- 与 Git 代码仓库有关的调试记录必须在对应的调试 Step 里写 Git 信息和完整 Git 可视化：仓库路径、分支、远端、关键 commit、推送状态、工作区状态、验证命令和 Mermaid 或 raw Git 图形都要写清楚，不能只放在代码同步记录或项目进度文件里，也不要单独漂成一个顶层 Git 章节。
- 硬件信息卡保持事实优先：总卡/型号卡的核心是设备索引，优先用 `.base`；具体设备卡的核心是 `# 硬件信息` 二列表。调试过程不要塞进硬件卡正文，必要时只作为可选补充链接到 `2_调试记录/`。
- 硬件信息卡正文模板以 Obsidian Templater 文件为准，当前机器路径是 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\硬件类型索引卡模板.md` 和 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板\2.1 Templater\具体硬件设备卡模板.md`。模板采用“通用核心字段 + 类型专属参数”方式，光谱仪、电机、相机、控制器、激光器、采集卡、电源、传感器等硬件都走同一套结构。
- 如果换新电脑、换 vault 位置，或上述硬件模板路径不存在，必须先向我确认当前 Obsidian 笔记库根目录、Templater 模板文件夹和硬件模板文件名；不要凭猜测在新位置重建模板。
- 每次新开项目工作记录时，都要先问是否需要代码拆解和项目理解笔记；如果暂时不需要，就只保留空的 `4_项目理解/` 文件夹。
- 图片是笔记的一等证据。用户提供轻量图片、截图、曲线图、设备照片或资料照片时，默认复制进 Obsidian 笔记库并在相关 Markdown 中直接嵌入，不只写文字摘要。
- 移动 Markdown 时要同步移动 `Attachments` 里的镜像附件目录。
- 旧总笔记改名时优先用 aliases 承接旧链接，不强行改写旧记录正文。
- 整理后检查普通双链和图片嵌入，目标是 `WIKILINK_MISSING=0`、`EMBED_MISSING_OUTSIDE_CODE=0`。
- 创建 Markdown note 时默认不在正文重复文件标题。
- 新项目不要先建一个孤立主笔记再慢慢长乱；一开始就建立 `0_总览`、`1_主线笔记`、`2_调试记录` 等骨架。
