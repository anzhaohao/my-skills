# azf-obsidian-project-record

这个 skill 用来维护安钊锋的 Obsidian 项目工作记录。它把一个项目整理成“总览、主线、调试记录、项目理解、补充资料”几层，让以后重新接手时能先看懂当前状态，再顺着主线理解每条记录之间的关系。硬件资产卡片由 `E:\software\Obsidian\安钊锋的外置大脑\03-Academic Toolkit\2_资源与档案\仪器设备资产` 手动维护，不属于本 skill 的写入范围。

## 什么时候触发

- 用户要求新开一个项目、建立项目总笔记、整理项目笔记、维护调试记录。
- 项目文件夹在 `01-Project` 下，需要重新拆分、编号、移动或同步链接。
- 用户提供实验截图、GUI 状态图、硬件状态图、报错、结果图，需要写入 Obsidian。
- 用户说“新项目也按这个方式记录”“同步调试记录”“帮我把记录整理清楚”。
- 用户希望新项目从一开始就达到 FrogTrace 这种“总览清楚、主线清楚、调试记录归位、图片不断链”的效果。

## 当前关键规则

- 首选结构：

```text
0_总览/
1_主线笔记/
2_调试记录/
3_项目理解/
4_补充资料/
```

当前参考项目是 `E:\software\Obsidian\安钊锋的外置大脑\01-Project\20260527_FrogTrace`：`0_总览` 放总笔记和项目索引，`1_主线笔记` 放问题主线，`2_调试记录` 放时间线证据，`3_项目理解` 放流程/源码理解，`4_补充资料` 放方案、评价和归档资料。

- 总笔记保持轻量，优先作为项目仪表盘；推荐顺序是：当前状态 → 当前主线 / 主线与当前状态 → 目前待解决的问题 / 需求 → 未归类调试记录 / 小 bug 收件箱。
- 新手阅读顺序、常用入口、文件夹说明应迁移到 `0_总览/项目名新手阅读顺序.md` 或补充说明里，不长期堆在总笔记。
- 总笔记正文不重复写文件标题；因为没有正文标题，一级内容标题从 `#` 开始，并加中文序号，例如 `# 一、现在先看这里`、`# 二、当前主线`、`# 三、目前待解决的问题`。
- 总笔记顶部维护 `目前待解决的问题 / 需求` 区域；表格只做轻量索引，不要 `入口` 列，列为：事项、等级、问题名称、状态。
- 事项编号统一用 `Q1`、`Q2`、`Q3`，方便在对话、提交信息和调试记录里直接引用；等级保留醒目的 `🚨 核心`、`⭐ 重要`、`🌱 后续`。
- 问题表格下方固定写一行 `等级可选：🚨 核心 / ⭐ 重要 / 🌱 后续`，方便后续复制调整。
- 问题的描述、当前判断、关联证据和下一步放在表格下面的短卡片里，避免横向表格太挤。
- 总笔记问题表的状态只有两个：`未创建主线`、`已有主线`；新建问题默认 `未创建主线`，已有主线后再改为 `已有主线`。
- 新增问题/需求必须调用当前 Vault 中动态发现的 Templater 命令或模板，只弹出三个输入：问题名称、问题描述、等级选择。总笔记问题表下方可放 Buttons 插件按钮，按钮必须调用当前发现的模板，不得在 skill 中写死动作名、路径或文件名。
- 主线笔记用来串联不同工作记录之间的关系，避免单看记录时看不出因果和前后变化。
- 所有项目记录相关操作（新建、补全、整理、同步、重构）都必须先在 `E:\software\Obsidian\安钊锋的外置大脑\05-Junk Drawer\2_模板` 下递归寻找匹配的 Templater/索引模板；找不到再检查当前 Vault 的 Templater 配置。调用模板后再填入从当前请求、agent-memory、已有笔记、代码仓库和必要的本地聊天记录中恢复的真实信息；不得手工重建模板正文。主线 frontmatter 必须包含 `当前状态: 一句话状态`，供总笔记主线索引读取；主线名称列由索引读取 `file.name`，不要另建 `主线` 属性。主线正文使用 `# 一句话说明`、`# 当前状态`、`# 推进记录`、`# 相关入口`。推进记录按日期标题写，不再用 Step 日记；每次推进下面跟随本次方案、调试记录和一句结论。代码状态、gitGraph、命令、日志和证据放进对应调试记录，不写进主线。
- 主线推进记录采用三档分流：一两句话能说清且无证据时写主线并标 `调试记录: 无单独记录`；一段话加说明性截图时用折叠 callout；有证据截图、实机结果、CSV、日志、命令输出、走弯路或未来可能被质疑的结论时必须开调试记录。
- 代码修改、调试、硬件排查、Git 工作和日志阅读期间，Obsidian 笔记库默认只读。只有我明确说“更新笔记、同步笔记、写入笔记、整理笔记”或确认一次笔记同步方案，才允许修改总览、主线或调试记录；仪器设备资产卡片始终由我手动维护。
- 如果代码/调试进展值得记录但我没有要求更新笔记，只在最后说明“笔记未更新，可后续单独同步”，不要顺手改笔记。
- 小 bug 不需要每个都升级成主线，但每个 bug/debug 记录都必须接入入口：先在 `2_调试记录/` 创建记录，并写 `归类状态: 未归类`；总笔记用“未归类调试记录 / 小 bug 收件箱”展示；接入主线后改为 `归类状态: 已归类` 并补 `关联主线`。
- “独立成文/单独成一个 Markdown”不等于“未归类”。只有用户明确说“独立记录”“独立调试记录”“单独记录”“不要归类”或直接要求不要接入主线时，才默认写 `归类状态: 未归类`；否则如果已找到相关主线，或者该记录已经被主线引用，就写 `归类状态: 已归类` 并补 `关联主线`。
- 如果 Obsidian Bases 可用，必须调用当前发现的 Base/Templater 模板生成视图，筛选 `项目名称 == 项目名` 且 `归类状态 == 未归类` 的记录，并嵌入总笔记；如果找不到匹配模板，先询问，不要手写 Base 内容或猜测旧模板。
- 调试记录放实际发生过的过程；目前 FrogTrace 调试记录只使用一种时间线格式：`# 一、调试目标`、`# 二、调试过程`、`# 三、调试结论`，过程里按真实时间顺序写 `Step 1`、`Step 2`、`Step 3`。
- 调试记录必须调用当前发现的 Obsidian/Templater 调试记录模板；skill 只保留记录规则，不内嵌模板正文，也不指定模板路径或文件名。
- 如果找不到匹配模板、Templater 无法调用或模板角色不明确，必须先向我确认；不要凭猜测在新位置重建模板。
- 不再为调试记录单独区分“实机验证记录”“代码修改记录”“测试表记录”等模板。真实硬件测试、代码修改、复测和日志排查都放进同一种时间线格式里。
- 与 Git 代码仓库有关的调试记录必须在对应的调试 Step 里写 Git 信息和完整 Git 可视化：仓库路径、分支、远端、关键 commit、推送状态、工作区状态、验证命令和 Mermaid 或 raw Git 图形都要写清楚，不能只放在代码同步记录或项目进度文件里，也不要单独漂成一个顶层 Git 章节。
- 硬件状态、设备测试和连接过程仍可写入 `2_调试记录/`，但硬件资产事实不在本 skill 中建卡或维护。
- 每次我说“新建项目到 Obsidian”时，先从 agent-memory、已有 Vault 笔记、代码仓库和必要的本地聊天记录恢复事实，再创建合适命名的项目文件夹；默认必须生成的内容笔记只有总笔记。`0_总览` 中的主线/未归类记录索引属于结构视图，按当前 Base/Components 模板保持存在，即使暂时为空也不生成占位卡。即使发现日志或截图，也只有在我明确要求导入/整理/生成实验或调试记录时，才调用相应内容模板。
- 图片是笔记的一等证据。用户提供轻量图片、截图、曲线图、设备照片或资料照片时，默认复制进 Obsidian 笔记库并在相关 Markdown 中直接嵌入，不只写文字摘要。
- 移动 Markdown 时要同步移动 `Attachments` 里的镜像附件目录。
- 旧总笔记改名时优先用 aliases 承接旧链接，不强行改写旧记录正文。
- 整理后检查普通双链和图片嵌入，目标是 `WIKILINK_MISSING=0`、`EMBED_MISSING_OUTSIDE_CODE=0`。
- 创建 Markdown note 时默认不在正文重复文件标题。
- 新项目不要先建一个孤立主笔记再慢慢长乱；创建项目文件夹时按当前 FrogTrace 骨架建立 `0_总览`、`1_主线笔记`、`2_调试记录`、`3_项目理解`、`4_补充资料`，但只生成有真实信息支撑的笔记。
